Summary Of The Article & Bug:

According to this article and the researchers at WordFence, this vulnerability is a Critical bug with a CVSS score of 9.8 out of 10. The bug is in the User Registration & Membership plugin, which is an extremely popular plugin that is used to manage user registrations and memberships on WordPress sites. This bug allows malicious actors to create admin accounts remotely without any authentication, which can lead to a complete takeover of the targeted site. The vulnerability is due to a flaw in the plugin's code that fails to properly validate user input, which allows attackers to exploit the registration process and grant themselves escalated privileges at account creation. This bug is reported to effect all versions of the plugin up to 5.1.2, and the developers have released a patch in the latest version.


Technical Breakdown Of Bug & Hypothetical Demo:


The technical details of this bug are quite interesting. Specifically, the plugin does not properly check the role assigned to a new user during registration, which allows attackers to specify an admin role when creating a new account. This means that an attacker can simply send a specially crafted registration request with the admin role specified, and the plugin will create an account with administrative privileges without any authentication. Once an attacker has created an admin account, they can log in and take complete control of the targeted WordPress site, including installing malicious plugins, modifying content, and stealing sensitive information. This vulnerability is dangerous because of how easy it is to exploit and I will now provide a theoretical example of how an attacker may escalate their privileges using this bug.

Hypothetical Reconnaissance Phase / Workflow:


For the purpose of this "demo", let us assume that we as the attacker (or researcher) have identified a potential target, which we will call nicks-demo-target.com, and we want to find out some info on the site including; whether or not it is indeed running WordPress, if the plugin exists on the site, what version of the plugin is running (keeping in mind that only versions up to and including 5.1.2 are vulnerable), and whether or not the site has frontend registration enabled. To identify if a site is infact running WordPress, we can use tools like; Wappalyzer or WhatCMS, which are both great tools but I tend to go the more simple and manual route for just a simple confirmation of Yes or No. I will typically just navigate to the site and add /wp-admin/ to the end of the URL and see if it redirects me to the admin login page or hits me with a 404. If it successfully redirects me to the login page, then I know for a definite fact that the site is built with WordPress. Regardless of how we confirm that the site is running WordPress, the next step is to identify if the vulnerable plugin is installed on the site. To do this, I highly recommend using the tool WPScan which is a powerful WordPress scanning tool that I have been utilizing abundantly over the course of the year and have had great success with. The command that we would use to scan for any vulnerable plugins detected on the site would be wpscan --url nicks-demo-target.com -e vp --plugins-detected mixed --api-token [API-TOKEN] --random-user-agent where the --url nicks-demo-target.com specifies the target site's URL, the -e vp flag tells WPScan to enumerate only the plugins on the site that have known vulnerabilities, the --plugins-detected mixed uses a compination of Passive and Agressive detection methods to find plugins, the --api-token [API-TOKEN] uses a specified API token to connect to the WPVulnDB and properly cross-reference the detected velnerabilities, and the --random-user-agent flag randomizes the user agent for the scan to help obfuscate the scan and make it less likely to be blocked by simple filters or WAFs. After running this scan, we would have a list of detected vulnerable plugins on the site, for the purpose of this "demo", lets move forward with the assumption that the scan found the User Registration & Membership plugin installed on the site and is indeed running a vulnerable version. This completes the "reconnaissance phase" and we can now move onto the next phase which covers how we could hypotetically exploit the bug.

Hypothetcial Enumeration & Exploitation:


Now onto how an attacker (or researcher) could hypothetically exploit this bug. To start, we would need to look for any exposed registration endpoints on the site. This can be done by using a tool like BurpSuite to intercept the traffic between our browser and the target site to look for any registration endpoints. Alternatively, we could also use a tool like FFUF to fuzz the site for common registration endpoints such as /register/ or /signup/ to see if we can find any paths that are utilized for user registration. For the purpose of this scenario, lets assume that we found a registration endpoint at nicks-demo-target.com/register/. Once we identify the registration enpoint, our next hypothetical step would be to send a normal registration request to the endpoint and capture it via BurSuite. This allows us (as an attacker or researcher) to see how the registration process works and what the request looks like. A normal registration request could look something like this:
POST /register/ HTTP/1.1
Host: nicks-demo-target.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Content-Type: application/x-www-form-urlencoded
Content-Length: 100

username=normaluser&
email=normaluser@normal.com&
role=subscriber&
password=basicPassword123!&
confirm_password=basicPassword123!&
form_id=4&
ur-frontend-form-id=4&
_wpnonce=abc123def456ghi789jkl012mno345pqr678stu901vwx234yz567890

After sending and intercepting this normal registration request, we can modify the request to specify an admin role instead of that of a subscriber. This is where the vulnerability comes into play, as the plugin fails to properly validate the role specified in the registration request we as the attacker can simply change the role=subscriber to role=administrator and send the request. The modified request would look something like this:
POST /register/ HTTP/1.1
Host: nicks-demo-target.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Content-Type: application/x-www-form-urlencoded
Content-Length: 100

username=hackerman&
email=hackerman@hacktheplanet.ru&
role=administrator&
password=superSecretPassword666!!!&
confirm_password=superSecretPassword666!!!&
form_id=4&
ur-frontend-form-id=4&
_wpnonce=abc123def456ghi789jkl012mno345pqr678stu901vwx234yz567890

After sending the modified request, the plugin would process it and create a new user account with admin privileges without any authentication, which would allow us as the "attacker" (or researcher) to head over to the admin login page that we identified earlier and log in with the newly created admin account to take complete control pf the site...it is really that simple...hypothetically of course.


Implications For WordPress Users:


The effects of this bug can be devestating for WordPress users on both the side of an admin / owner of an affected site and on the side of a regular user who has an account on an affected site. For the owner of a site that has been exploited by this bug, they could potentially lose all of their data, have their site defaced, have malicious plugins installed on their site, and even have their site used as a launchpad for further attacks on other sites. This is because the attacker has full control over the site and can do anything that any other administrator can do. For regular or "subscriber" level users on an affected site, they could have their personal information stolen, have their accounts used to post spam / malicious content, and even have their accounts deleted by the attacker.
